Arquitectura del plugin, estructura de archivos, modelo de datos, conectores y configuración.

Arquitectura del plugin

lgaas-activation es un plugin de Claude Cowork empaquetado como archivo .plugin (formato zip). Contiene 11 skills independientes que se invocan secuencialmente para ejecutar el pipeline completo de Lead Generation as a Service.

Cada skill es autónoma: tiene su propio SKILL.md con instrucciones, un Activation Protocol (flujo guiado con preguntas de múltiple elección), y archivos de referencia opcionales en su carpeta references/ y assets/.

El plugin no requiere MCP servers propios — se conecta a los MCPs ya disponibles en el entorno Cowork del usuario (Apollo, Clay, LemList, Google Calendar, Gmail).

Reglas de validación al subir el plugin a una organización:

name — kebab-case, minúsculas con guiones. Debe coincidir con la carpeta raíz.
version — Formato semver: MAJOR.MINOR.PATCH (ej: 1.2.0)
description — Máximo 500 caracteres
keywords — Array de strings para búsqueda. Sin límite.
Anatomía de un SKILL.md
Cada skill tiene un archivo SKILL.md con dos partes: el YAML frontmatter (metadatos) y el cuerpo Markdown (instrucciones para Claude).
--- # YAML Frontmatter (obligatorio)
name: 3-lead-enrichment            # Debe coincidir con carpeta
description: >                     # Máx 1024 caracteres
  Step 3 of the LGaaS Activation pipeline. Takes the
  consultant's selected hipersegment and generates a
  branded base de datos .docx...
  Trigger with: "Activa Lead Enrichment", ...
version: 1.0.0                     # Semver
---

# Cuerpo Markdown (instrucciones para Claude)

## Model Routing                   # Qué modelo usar y cuándo
## Activation Protocol             # Flujo guiado con AskUserQuestion
## Phase 1 — [Nombre]              # Pasos de ejecución
## Phase 2 — [Nombre]
...
## Relationship with other skills  # Dependencias y handoffs
ⓘ Reglas YAML: Solo se permiten name, description y version en el frontmatter. Campos como model, capabilities, dependencies o tags causarán error de validación al subir.
ⓘ Descripción: Debe incluir qué hace la skill + triggers de activación (frases que la disparan). Máx 1024 caracteres. El sistema usa este texto para decidir cuándo activar la skill — sé explícito con los triggers.
Patrón de Activation Protocol
Cada skill tiene una sección ## Activation Protocol que define cómo Claude debe guiar al consultor cuando la skill se activa. Este es el patrón estándar:
## Activation Protocol

**Trigger:** "Activa Lead Enrichment", "Step 3", ...

### Step 1 — Confirmar preparación
Usar AskUserQuestion:
> "Perfecto, vamos a generar la base de datos."
> Opciones:
> - A) ¿Datos listos? (RECOMENDADO)
> - B) Necesito revisar algo

### Step 2 — Estimación de créditos
Usar AskUserQuestion:
> "Estimación de créditos: Apollo ~120, Clay ~60..."
> Opciones:
> - A) Proceder con todo (RECOMENDADO)
> - B) Solo Apollo
> - C) Solo top 20

### Step 3 — ¿Arrancamos?
> Confirmar y lanzar ejecución
Convenciones:
• Opciones con letras: A), B), C)...
• Marcar la opción sugerida con (RECOMENDADO)
• Idioma: español siempre (el consultor opera en español)
• Usar AskUserQuestion de Cowork para cada paso interactivo
• El último paso siempre es confirmación antes de ejecutar
Model routing

Cada skill especifica qué modelo de IA usar en cada fase. La regla general:

Opus 4.6
Tareas estratégicas, creativas y de análisis: planificación de hipersegmentos, diseño de secuencias, redacción de copy, manejo de objeciones, scoring.
Sonnet
Tareas de ejecución y API: llamadas a Apollo/Clay/LemList, procesamiento de datos, generación de Excel, lectura de documentos, web scraping.

Esto se especifica en cada SKILL.md con la directiva model: "opus" o model: "sonnet" dentro de las instrucciones (no en el YAML frontmatter).

Integración con SABI

SABI no tiene API — la exportación es manual por parte del consultor. El plugin lo gestiona así:

1. En el Step 1 (Market Segmentation), Claude genera los filtros exactos: códigos CNAE, rango de empleados, rango de facturación, columnas a exportar.
2. El consultor va a SABI, aplica los filtros, exporta como .xlsx.
3. Sube el .xlsx a la sesión de Cowork.
4. Claude cruza los datos de SABI con Apollo para generar el TAM definitivo con cobertura de Decision Makers.

Si el mercado objetivo no es España/Portugal, SABI se omite y se usa solo Apollo.

Cascada de verificación de email

En el Step 3 (Lead Enrichment), los emails de cada Decision Maker se verifican en 3 niveles:

Apollo
People Match
→
Clay
Contact Enrich
→
LemList
Find & Verify

Si un email pasa las 3 verificaciones, se marca como verified. Si solo pasa 1-2, se marca como risky. Si no pasa ninguna, se excluye del envío.

Se configuran en el Activation Protocol de cada skill. El consultor puede elegir ejecución manual en vez de programada.

Actualizar o modificar el plugin

Para modificar una skill o añadir funcionalidad:

1. Descomprime el .plugin (es un zip estándar)
2. Edita el SKILL.md correspondiente
3. Incrementa la version en plugin.json
4. Re-empaqueta: zip -r lgaas-activation.plugin .claude-plugin/ skills/
5. Sube de nuevo a la organización

También puedes pedir a Claude en Cowork: “Modifica la skill 6-message-writer para añadir WhatsApp” y usar la skill skill-creator para regenerar y reempaquetar.
